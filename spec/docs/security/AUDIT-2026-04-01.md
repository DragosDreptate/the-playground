### A2. Problèmes identifiés

#### [A-MOYE-1] Contournement du flux de publication via `updateMomentAction`

- **Sévérité** : MOYEN
- **Fichier** : `src/app/actions/moment.ts:219`
- **Description** : `updateMomentAction` passe `status` au usecase `updateMoment` sans restriction de transition. Un HOST peut envoyér `status: "PUBLISHED"` et contourner le usecase `publishMoment`, bypassant les emails de notification aux membres.
- **Vecteur** : FormData avec `status: "PUBLISHED"` envoyé depuis un client modifié.
- **Impact** : bug de logique métier, pas une vulnérabilité de sécurité. Seul le HOST de son propre Circle est concerne — il publie son propre événement mais ses membres ne reçoivent pas la notification. Pas d'escalade de privilèges, pas de fuite de données, pas de vecteur d'attaque externe.
- **Correction** : filtrer `status` dans `updateMomentAction` pour n'autorisér que `CANCELLED`, ou ajouter une garde de transition dans le usecase `updateMoment`.
- **Statut** : [ ] À corriger

#### [A-INFO-1] Race conditions non-atomiques (acceptées MVP)

- **Sévérité** : INFO
- **Fichiers** : `join-moment.ts`, `handle-payment-webhook.ts`
- **Description** : `countByMomentIdAndStatus` + `create` ne sont pas atomiques. Fenetre de collision infime. La contrainte unique DB `(momentId, userId)` constitue un filet de sécurité.
- **Décision** : accepté pour le MVP, à surveiller sous charge.
- **Statut** : [x] Accepte (risque résiduel documenté)

- **Constat** : le middleware ne vérifie pas les sessions. La protection des routes est déléguée aux layouts RSC (`getCachedSession()` + `redirect()`).

- **Matcher** : exclut `/api/*` — les API routes doivent s'auto-protégér.
- **Statut** : [ ] À évaluer (ajout d'un guard middleware global vs statu quo)

## Points forts

L'audit a identifié de nombreux points positifs :

- **Architecture hexagonale solide** — séparation domaine/infra respectée, 147 tests de sécurité verts
- **RBAC complet** — tous les usecases vérifient l'autorisation (HOST vs PLAYER, ownership)
- **Isolation multi-tenant** — le `circleId` est toujours issu de la DB, jamais de l'input utilisateur
- **Webhooks bien sécurisés** — Stripe (signature lib officielle) + Sentry (HMAC SHA-256 + `timingSafeEqual`)
- **Protection open redirect** — `safeCallbackUrl()` correctement implémentée
- **HSTS + headers de sécurité** — configurés avec preload
- **Database sessions** (pas JWT) — invalidation serveur possible
- **Source maps supprimées** apres upload Sentry
- **Suppression de compte** — hard delete avec cascadecomplète
- **DB en UE** (Neon eu-central-1) + PostHog EU
- **Cron secrets** — endpoints cron protégés par Bearer token
- **Bot blocking** — middleware bloqué les bots connus
- **OAuth state/PKCE** — géré automatiquement par Auth.js v5
- **SSRF protection** — `cover-image.ts` utilise une allowlist de domaines
